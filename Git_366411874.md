# Git fundamentals, a complete guide

f you already work with <a href="https://git-scm.com/book/en/v2" class="external-link" rel="nofollow"><u>Git</u></a> daily but want to have a good comprehension of Git **fundamentals**, then this post is for you.

Here, you'll have the chance to truly understand the **Git architecture** and how commands such as **add, checkout, reset, commit, merge, rebase, cherry-pick, pull, push** and **tag** work internally.

*Don't let Git master you*, learn the Git fundamentals and **master Git instead**.

Brace yourselves, a **complete guide about Git** is about to start.

`이미 매일 Git을 사용하고 있지만 Git 기본 사항을 잘 이해하고 싶다면 이 게시물이 적합합니다.`

`여기에서 Git 아키텍처와 add, checkout, reset, commit, merge, rebase, cherry-pick, pull, push 및 tag와 같은 명령이 내부적으로 작동하는 방식을 진정으로 이해할 수 있습니다.`

`Git이 당신을 마스터하도록 두지 말고 Git 기본 사항을 배우고 대신 Git을 마스터하십시오.`

`힘내세요. Git에 대한 전체 가이드가 곧 시작됩니다.`

The Git CLI provides two types of commands:

- **plumbing**, which consists of *low-level commands* used by Git internally behind the scenes when users type *high-level* commands

- **porcelain**, which are the *high-level* commands commonly used by Git users

In this guide, we'll see how the **plumbing commands relate to the porcelain commands** that we use day-to-day.

`Git CLI는 두 가지 유형의 명령을 제공합니다.`

- `사용자가 높은 수준의 명령을 입력할 때 내부적으로 Git이 사용하는 낮은 수준의 명령으로 구성된 배관`

## The Object Database

Using the UNIX tool `find` we can see the structure of the `.git/objects` folder:

`UNIX 도구 찾기를 사용하여 .git/objects 폴더의 구조를 볼 수 있습니다.`

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ find .git/object

.git/objects
.git/objects/pack
.git/objects/info
```

</div>

</div>

In Git, everything is persisted in the `.git/objects` structure, which is the **Git Object Database**.

What kind of content can we persist in Git? *Everything*.

> **Wait!**
>
> How is that possible?

Through the use of <a href="https://en.wikipedia.org/wiki/Hash_function" class="external-link" rel="nofollow"><u>hash functions</u></a>.

`Git에서는 모든 것이 Git 개체 데이터베이스인 .git/objects 구조에 유지됩니다.`

`Git에서 어떤 종류의 콘텐츠를 유지할 수 있습니까? 모든 것.`

`해시 함수 사용을 통해.`

### Hashing for the rescue

A **hash function** *maps data of arbitrary, dynamic size into fixed-size values*. By doing this, we can store/persist anything because the final value will have always the *same size*.

Bad implementations of hash functions can easily lead to **collisions**, where two different dynamic-size data could map to the same final hash of fixed-size.

<a href="https://en.wikipedia.org/wiki/SHA-1" class="external-link" rel="nofollow"><u>SHA-1</u></a> is a well-known implementation of the hash function that is in general safe and hardly has collisions.

Let's take, for instance, the hashing of the string `my precious`:

`해시 함수는 임의의 동적 크기의 데이터를 고정 크기 값으로 매핑합니다. 이렇게 하면 최종 값이 항상 같은 크기를 갖기 때문에 무엇이든 저장/유지할 수 있습니다.`

`해시 함수의 잘못된 구현은 쉽게 충돌로 이어질 수 있으며, 여기서 두 개의 서로 다른 동적 크기 데이터가 고정 크기의 동일한 최종 해시에 매핑될 수 있습니다.`

`SHA-1은 일반적으로 안전하고 충돌이 거의 없는 해시 함수의 잘 알려진 구현입니다.`

`예를 들어 문자열 my famous의 해싱을 살펴보겠습니다.`

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ echo -e "my precious" | openssl sha1
fa628c8eeaa9527cfb5ac39f43c3760fe4bf8bed
```

</div>

</div>

*Note: If you're using Linux, you can use the command* `sha1sum` instead of `OpenSSL`.

`참고: Linux를 사용하는 경우 OpenSSL 대신 sha1sum 명령을 사용할 수 있습니다.`

### 🔵 Comparing differences in the content

A *good hashing* is a safe practice where **we can't know the raw value**, i.e doing the reverse engineering.

In case we want to know *if the value has changed*, we just wrap the value into the hashing function and *voilà*, we can compare the difference:

`좋은 해싱은 원시 값을 알 수 없는 경우, 즉 리버스 엔지니어링을 수행하는 안전한 방법입니다.`

`값이 변경되었는지 알고 싶은 경우 값을 해싱 ​​함수로 래핑하면 차이를 비교할 수 있습니다.`

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ echo -e "my precious" | openssl sha1
fa628c8eeaa9527cfb5ac39f43c3760fe4bf8bed

$ echo -e "no longer my precious" | openssl sha1
2e71c9ae2ef57194955feeaa99f8543ea4cd9f9f
```

</div>

</div>

If the *hashes are different*, then we can assume that the **value has changed**.

Can you spot an opportunity here? What about using **SHA-1** to store data and *just keep track of everything* by comparing hashes?

That's exactly what Git does internally 🤯.

`해시가 다르면 값이 변경되었다고 가정할 수 있습니다.`

`여기서 기회를 찾을 수 있습니까? SHA-1을 사용하여 데이터를 저장하고 해시를 비교하여 모든 것을 추적하는 것은 어떻습니까?`

`그것이 바로 Git이 내부적으로 하는 일입니다 🤯`

### 🔵 Git and SHA-1

**Git uses SHA-1 to generate hashing** of everything and stores it in the `.git/objects` folder. *Simple like that!*

The **plumbing** command `hash-object` does the job:

`Git은 SHA-1을 사용하여 모든 항목의 해싱을 생성하고 .git/objects 폴더에 저장합니다. 그렇게 간단합니다!`

`배관 명령 hash-object가 작업을 수행합니다.`

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ echo "my precious" | git hash-object --stdin
8b73d29acc6ae79354c2b87ab791aecccf51701f
```

</div>

</div>

Let's compare with the `OpenSSL` version:

`OpenSSL 버전과 비교해 보겠습니다.`

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ echo -e "my precious" | openssl sha1
fa628c8eeaa9527cfb5ac39f43c3760fe4bf8bed
```

</div>

</div>

**Oooops**...it's quite different. That's because Git *prepends a specific word* followed by the *content size* and the delimiter `\0`. Such a word is what Git calls **the object type**.

Yes, *Git objects have types*. The first one we'll look into is **the blob object**.

`으아아아아아아아아아아아아아아아아아아. Git은 콘텐츠 크기와 구분 기호 \0이 뒤에 오는 특정 단어를 앞에 추가하기 때문입니다. 이러한 단어는 Git이 개체 유형이라고 부르는 것입니다.`

`예, Git 객체에는 유형이 있습니다. 우리가 살펴볼 첫 번째 항목은 blob 개체입니다.`

### 🔵 The blob object

When we send for instance the string "my precious" to the `hash-object` command, Git prepends the pattern `{object_type} {content_size}\0` to the SHA-1 function, so that:

`예를 들어 우리가 해시 객체 명령에 문자열 "나의 소중한"을 보낼 때, Git은 {object_type} {content_size}\0 패턴을 SHA-1 함수에 추가하여 다음을 수행합니다.`

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
blob 12\0myprecious
```

</div>

</div>

Then:

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ echo -e "blob 12\0my precious" | openssl sha1
8b73d29acc6ae79354c2b87ab791aecccf51701f

$ echo "my precious" | git hash-object --stdin
8b73d29acc6ae79354c2b87ab791aecccf51701f
```

</div>

</div>

**Yay!** 🎉

### 🔵 Storing blobs in the database

But the command `hash-object` itself does not persist into the `.git/objects` folder. We should append the option `-w` and the object will be persisted:

`그러나 hash-object 명령 자체는 .git/objects 폴더에 유지되지 않습니다. -w 옵션을 추가해야 개체가 유지됩니다.`

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ echo "my precious" | git hash-object --stdin -w
8b73d29acc6ae79354c2b87ab791aecccf51701f

$ find .git/objects
...
.git/objects/8b
.git/objects/8b/73d29acc6ae79354c2b87ab791aecccf51701f

### Or, simply
$ find .git/objects -type f
.git/objects/8b/73d29acc6ae79354c2b87ab791aecccf51701f
```

</div>

</div>

### Reading the raw content of a blob

We already know that for cryptographic reasons it's not possible to read the raw content from its hashing version.

> 🤔 Ok, but wait.
>
> How does Git *get to know the original value*?

It uses the hash as a **key pointing to a value**, which is the *original content itself* using a compression algorithm called <a href="https://en.wikipedia.org/wiki/Zlib" class="external-link" rel="nofollow"><u>Zlib</u></a>, that **compacts the content** and stores it in the object database, hence **saving storage space**.

`우리는 암호화상의 이유로 해싱 버전에서 원시 콘텐츠를 읽을 수 없다는 것을 이미 알고 있습니다.`

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
알겠습니다. 하지만 기다리세요.

Git은 원래 값을 어떻게 알 수 있습니까?
```

</div>

</div>

`Zlib라는 압축 알고리즘을 사용하여 원본 콘텐츠 자체인 값을 가리키는 키로 해시를 사용하여 콘텐츠를 압축하고 개체 데이터베이스에 저장하므로 저장 공간이 절약됩니다.`

The **plumbing** command `cat-file` does the job so that, *given a key*, inflates the compressed data thus getting the original content:

`배관 명령 cat-file은 키가 주어지면 압축된 데이터를 부풀려 원본 콘텐츠를 가져오는 작업을 수행합니다.`

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ git cat-file -p 8b73d29acc6ae79354c2b87ab791aecccf51701f
my precious
```

</div>

</div>

In case you are guessing, that's right, **Git is a key-value database**!

`추측하는 경우에 맞습니다. Git은 키-값 데이터베이스입니다!`

### 🔵 Promoting blobs

When using Git, we want to work on the content and **share them with other people**.

Commonly, after working on various files/blobs, we are ready to *share them and sign our names for the final work*.

In other words, we need to **group, promote and add metadata to our blobs.** This process works as follows:

1.  Add the blob to a *staging area*

2.  Group **all blobs** in the stage area into a *tree structure*

3.  Add **metadata** to the *tree structure* (author name, date, a semantic message)

Let's see the *above steps in detail*.

`Git을 사용할 때 콘텐츠를 작업하고 다른 사람들과 공유하고 싶습니다.`

`일반적으로 다양한 파일/블롭에서 작업한 후 최종 작업을 위해 이를 공유하고 서명할 준비가 되었습니다.`

`즉, Blob에 메타데이터를 그룹화, 승격 및 추가해야 합니다. 이 프로세스는 다음과 같이 작동합니다.`

1.  `준비 영역에 Blob 추가`

2.  `스테이지 영역의 모든 Blob을 트리 구조로 그룹화`

3.  `트리 구조에 메타데이터 추가(저자 이름, 날짜, 의미론적 메시지)`

`위의 단계를 자세히 살펴보겠습니다.`

### 🔵 Stage area, the index

The **plumbing** command `update-index` allows to add a blob to the stage area and give a name to it:

`배관 명령 update-index를 사용하면 스테이지 영역에 BLOB를 추가하고 이름을 지정할 수 있습니다.`

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ git update-index \
    --add \
    --cacheinfo 100644 \
    8b73d29acc6ae79354c2b87ab791aecccf51701f \
    index.txt
```

</div>

</div>

- `--add`: adds the blob to the stage, also called **the index**

- `--cacheinfo`: used to register a file that is not in the working directory yet

- the blob hash

- `index.txt`: a name for the blob in the index

<!-- -->

- --add: 인덱스라고도 하는 blob을 스테이지에 추가합니다.

- `--cacheinfo: 아직 작업 디렉토리에 없는 파일을 등록하는 데 사용`

- `얼룩 해시`

- `index.txt: 인덱스에 있는 Blob의 이름`

Where does Git *store the index*?

`Git은 인덱스를 어디에 저장합니까?`

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

It's not human-readable though, it's compressed using Zlib.

`그러나 사람이 읽을 수는 없으며 Zlib를 사용하여 압축됩니다.`

예를 들어 원하는 만큼 인덱스에 BLOB를 추가할 수 있습니다.

After adding blobs to the index, we can group them into a **tree structure** which is ready to be promoted.

`Blob을 인덱스에 추가한 후 승격할 준비가 된 트리 구조로 그룹화할 수 있습니다.`

### 🔵 The tree object

When using the **plumbing** command `write-tree`, **Git groups all blobs that were added to the index** and create another object in the `.git/objects` folder:

`배관 명령 write-tree를 사용할 때 Git은 인덱스에 추가된 모든 Blob을 그룹화하고 .git/objects 폴더에 다른 개체를 만듭니다.`

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ git write-tree
3725c9e313e5ae764b2451a8f3b1415bf67cf471
```

</div>

</div>

Checking the `.git/objects` folder, note that a new object was created:

`.git/objects 폴더를 확인하면 새 객체가 생성되었음을 알 수 있습니다.`

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ find .git/objects

### The new object
.git/objects/37
.git/objects/37/25c9e313e5ae764b2451a8f3b1415bf67cf471

### The blob previously created
.git/objects/8b
.git/objects/8b/73d29acc6ae79354c2b87ab791aecccf51701f
```

</div>

</div>

Let's retrieve the original value using `cat-file` to understand better:

<div class="code panel pdl" style="border-width: 1px;">

``` syntaxhighlighter-pre
$ git reset b77b
```

</div>

</div>

Checking with `git status`:

<div class="codeContent panelContent pdl">

*The revision commit was changed in the* ***fix branch*** *and all the differences were* ***moved to the index***.

Still, what should we do in case we want to **reset AND discard** all the differences? Just passing on the option `--hard`:

By using `git reset --hard`, any difference between revisions **will be discarded** and they *won't appear in the index*.

### 💡 Golden tip about moving a branch

In case we want to perform the *plumbing* `update-ref` on another branch, there's no need to checkout the branch like needed in **git reset**.

We can perform the **porcelain** `git branch -f source target` instead:

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ git branch -f main b77b
```

</div>

</div>

Under the hood, it performs a `git reset --hard` in the source branch. Let's check to which commit the **main branch** is pointing:

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ git log main --pretty=oneline -n1
b77b454a9a507f839880879a895ac4f241177a28 (main) another commit
```

</div>

</div>

Also, we confirm that the **fix branch** is still pointing to the `369cd` commit:

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ git log fix --pretty=oneline -n1
369cd96b1f1ef6fa7de1ff2ed12e15be979dcffa (HEAD -> fix, test) add files
```

</div>

</div>

We did a "git reset" *without moving the HEAD*!

Not rare, *instead of moving a branch pointer*, we want to **apply a specific commit to the current branch**.

Meet **cherry-pick**.

### 🔵 git cherry-pick

With the **porcelain** `git cherry-pick`, we can apply an arbitrary commit to the current branch.

Take the following scenario:

<span class="confluence-embedded-file-wrapper image-center-wrapper"><img src="./media/attachments/366411874/383778939.png" class="confluence-embedded-image image-center" loading="lazy" data-image-src="attachments/366411874/383778939.png" data-height="253" data-width="880" data-unresolved-comment-count="0" data-linked-resource-id="383778939" data-linked-resource-version="1" data-linked-resource-type="attachment" data-linked-resource-default-alias="i0n7mldmwq8p3be6bzlf.png" data-base-url="https://never4got10.atlassian.net/wiki" data-linked-resource-content-type="image/png" data-linked-resource-container-id="366411874" data-linked-resource-container-version="2" data-media-id="d0f489ae-be87-4f4f-9350-d425fc4f2517" data-media-type="file" alt="scenario 42" /></span>

- **main** points to C3 - C2 - C1

- **fix** points to C5 - C4 - C2 - C1

- **HEAD** points to fix

In the *fix branch*, we are **missing the C3 commit**, which is being referenced by the *main branch*.

We can apply it by running `git cherry-pick C3`:

<span class="confluence-embedded-file-wrapper image-center-wrapper"><img src="./media/attachments/366411874/383778945.png" class="confluence-embedded-image image-center" loading="lazy" data-image-src="attachments/366411874/383778945.png" data-height="475" data-width="880" data-unresolved-comment-count="0" data-linked-resource-id="383778945" data-linked-resource-version="1" data-linked-resource-type="attachment" data-linked-resource-default-alias="uvw4m2fom52ggxvn0o13.png" data-base-url="https://never4got10.atlassian.net/wiki" data-linked-resource-content-type="image/png" data-linked-resource-container-id="366411874" data-linked-resource-container-version="2" data-media-id="73b5999b-e970-479f-bdcc-704e0f5cbc73" data-media-type="file" alt="cherry pick A" /></span>

Note that:

- the **C3** commit will be cloned into a new commit called **C3'**

- this new commit will reference the C5 commit

- **fix will move the pointer to C3'**

- *HEAD keeps pointing to fix*

After applying changes, the graph will be represented as follows:

There's another way to move the pointer of a branch though. It consists of *applying an arbitrary commit of another branch* but **merging the differences** if needed.

You're not wrong, we're talking about **git merge** here.

### 🔵 git merge

Let's describe the following scenario:

<span class="confluence-embedded-file-wrapper image-center-wrapper"><img src="./media/attachments/366411874/383778957.png" class="confluence-embedded-image image-center" loading="lazy" data-image-src="attachments/366411874/383778957.png" data-height="284" data-width="880" data-unresolved-comment-count="0" data-linked-resource-id="383778957" data-linked-resource-version="1" data-linked-resource-type="attachment" data-linked-resource-default-alias="61lv3lp1bsi3o7ingaxq.png" data-base-url="https://never4got10.atlassian.net/wiki" data-linked-resource-content-type="image/png" data-linked-resource-container-id="366411874" data-linked-resource-container-version="2" data-media-id="097f6de1-d6f1-40c0-afd5-bda0f7963616" data-media-type="file" alt="another scenario" /></span>

- main points to C3 - C2 - C1

- fix points to C4 - C3 - C2 - C1

- HEAD points to the main

We want to apply the fix branch into the current (main) branch, *a.k.a* perform a **git merge fix**.

Please note that the **fix branch contains all commits belonging to the main branch** (C3 - C2 - C1), having only *one commit ahead* of the main (C4).

In this case, the main branch will be "forwarded", pointing to the same commit as the fix branch.

This kind of merge is called **fast-forward**, as described in the image below:

### When fast-forward is not possible

Sometimes, our tree structure current's state does not allow fast-forward. Take the scenario below:

<span class="confluence-embedded-file-wrapper image-center-wrapper"><img src="./media/attachments/366411874/383778969.png" class="confluence-embedded-image image-center" loading="lazy" data-image-src="attachments/366411874/383778969.png" data-height="366" data-width="880" data-unresolved-comment-count="0" data-linked-resource-id="383778969" data-linked-resource-version="1" data-linked-resource-type="attachment" data-linked-resource-default-alias="a4tsezqc59orz8a2o2nb.png" data-base-url="https://never4got10.atlassian.net/wiki" data-linked-resource-content-type="image/png" data-linked-resource-container-id="366411874" data-linked-resource-container-version="2" data-media-id="57552718-eb5d-4a92-83d8-608252d3bd4f" data-media-type="file" alt="scenario 44" /></span>

As such, *fast-forward is not possible*.

However, for the merge to succeed, Git performs a technique called **Snapshotting**, composed of the following steps.

First, Git looks to the next **common parent** of the two branches, in this example, the **C2** commit.

Secondly, Git takes a **snapshot of the target** C3 commit branch:

<span class="confluence-embedded-file-wrapper image-center-wrapper"><img src="./media/attachments/366411874/383778981.png" class="confluence-embedded-image image-center" loading="lazy" data-image-src="attachments/366411874/383778981.png" data-height="507" data-width="880" data-unresolved-comment-count="0" data-linked-resource-id="383778981" data-linked-resource-version="1" data-linked-resource-type="attachment" data-linked-resource-default-alias="r5b784w4wejqx7xawynu.png" data-base-url="https://never4got10.atlassian.net/wiki" data-linked-resource-content-type="image/png" data-linked-resource-container-id="366411874" data-linked-resource-container-version="2" data-media-id="a465b250-26f1-4c63-bc9a-e0ab08269b2f" data-media-type="file" alt="target snapshot" /></span>

*Make no mistake*, this merge process is called the **three-way merge**!

Next, let's explore *another merge technique* where fast-**forward is not possible**, but instead of snapshotting and automatic commit merge, Git applies the differences **on top of the source branch**.

Yes, that's the **git rebase**.

### 🔵 git rebase

Consider the following image:

<span class="confluence-embedded-file-wrapper image-center-wrapper"><img src="./media/attachments/366411874/383779005.png" class="confluence-embedded-image image-center" loading="lazy" data-image-src="attachments/366411874/383779005.png" data-height="405" data-width="880" data-unresolved-comment-count="0" data-linked-resource-id="383779005" data-linked-resource-version="1" data-linked-resource-type="attachment" data-linked-resource-default-alias="n6u9hkcopd56anjjt6s7.png" data-base-url="https://never4got10.atlassian.net/wiki" data-linked-resource-content-type="image/png" data-linked-resource-container-id="366411874" data-linked-resource-container-version="2" data-media-id="35edabe2-3af5-4d03-9ebe-a022829fc2db" data-media-type="file" alt="scenario 55" /></span>

- fix points to C5 - C4 - C2 - C1

- HEAD points to fix

We want to **rebase** the main branch into the fix branch, by issuing `git rebase main`. But how does *git rebase* work?

👉 **git reset**

First, Git performs a **git reset main**, where the fix branch will point to the same main branch pointer: *C3 - C2 - C1*.

<span class="confluence-embedded-file-wrapper image-center-wrapper"><img src="./media/attachments/366411874/383779011.png" class="confluence-embedded-image image-center" loading="lazy" data-image-src="attachments/366411874/383779011.png" data-height="482" data-width="880" data-unresolved-comment-count="0" data-linked-resource-id="383779011" data-linked-resource-version="1" data-linked-resource-type="attachment" data-linked-resource-default-alias="qq56zo5l7v6urjib67n0.png" data-base-url="https://never4got10.atlassian.net/wiki" data-linked-resource-content-type="image/png" data-linked-resource-container-id="366411874" data-linked-resource-container-version="2" data-media-id="a8231072-3a65-40d9-911c-12e5539e8588" data-media-type="file" /></span>

At this moment, the C5 - C4 commits *have no references*.

👉 **git cherry-pick**

Second, Git performs a **git cherry-pick C5** into the current branch:

<span class="confluence-embedded-file-wrapper image-center-wrapper"><img src="./media/attachments/366411874/383779017.png" class="confluence-embedded-image image-center" loading="lazy" data-image-src="attachments/366411874/383779017.png" data-height="494" data-width="880" data-unresolved-comment-count="0" data-linked-resource-id="383779017" data-linked-resource-version="1" data-linked-resource-type="attachment" data-linked-resource-default-alias="ucbxeoflzk8izrb2w621.png" data-base-url="https://never4got10.atlassian.net/wiki" data-linked-resource-content-type="image/png" data-linked-resource-container-id="366411874" data-linked-resource-container-version="2" data-media-id="cc40fb6f-7cc4-478a-9ccb-ac17e81d5319" data-media-type="file" /></span>

## 🌐 Remote Branches

To work with remote branches, we have to *add a remote* to our local repository, using the **porcelain** command `git remote`.

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ git remote add origin git@github.com/myaccount/myrepo.git
```

</div>

</div>

Remotes are located in the `.git/refs/remotes` folder:

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ find .git/refs
...
.git/refs/remotes/origin
.git/refs/remotes/origin/main
```

</div>

</div>

### 🔵 Download from remote

How do we **synchronize** the remote branch with our local branch?

Git provides **two steps**:

👉 **git fetch**

By using the **porcelain** `git fetch origin main`, Git will download the remote branch and synchronize it with a new local branch called **origin/main**, also known as the **upstream branch**.

<span class="confluence-embedded-file-wrapper image-center-wrapper"><img src="./media/attachments/366411874/383779035.png" class="confluence-embedded-image image-center" loading="lazy" data-image-src="attachments/366411874/383779035.png" data-height="481" data-width="880" data-unresolved-comment-count="0" data-linked-resource-id="383779035" data-linked-resource-version="1" data-linked-resource-type="attachment" data-linked-resource-default-alias="9hii3cz4cjttk4pqnwzx.png" data-base-url="https://never4got10.atlassian.net/wiki" data-linked-resource-content-type="image/png" data-linked-resource-container-id="366411874" data-linked-resource-container-version="2" data-media-id="beec9784-2537-4ded-a5c0-aefeae28a43a" data-media-type="file" alt="fetch" /></span>

👉 **git merge**

After fetching and syncing the upstream branch, we can perform a `git merge origin/main` and because the upstream is ahead of our local branch, Git will safely apply a **fast-forward merge**.

<span class="confluence-embedded-file-wrapper image-center-wrapper"><img src="./media/attachments/366411874/383779041.png" class="confluence-embedded-image image-center" loading="lazy" data-image-src="attachments/366411874/383779041.png" data-height="513" data-width="880" data-unresolved-comment-count="0" data-linked-resource-id="383779041" data-linked-resource-version="1" data-linked-resource-type="attachment" data-linked-resource-default-alias="86x06eev8316smdlz8i2.png" data-base-url="https://never4got10.atlassian.net/wiki" data-linked-resource-content-type="image/png" data-linked-resource-container-id="366411874" data-linked-resource-container-version="2" data-media-id="97a7cd16-582e-4253-afee-91a944842f32" data-media-type="file" alt="fetch merge ff" /></span>

However, ***fetch + merge*** *could be repetitive*, as we would synchronize local/remote branches multiple times a day.

But today is *our lucky day*, and Git provides the **git pull** porcelain command, that performs fetch + merge on our behalf.

👉 **git pull**

With `git pull`, Git will perform fetch (synchronize remote with the upstream branch), and then merge the upstream branch into the local branch.

<span class="confluence-embedded-file-wrapper image-center-wrapper"><img src="./media/attachments/366411874/383779047.png" class="confluence-embedded-image image-center" loading="lazy" data-image-src="attachments/366411874/383779047.png" data-height="508" data-width="880" data-unresolved-comment-count="0" data-linked-resource-id="383779047" data-linked-resource-version="1" data-linked-resource-type="attachment" data-linked-resource-default-alias="ce7lb2f2bxpeoigfqbyr.png" data-base-url="https://never4got10.atlassian.net/wiki" data-linked-resource-content-type="image/png" data-linked-resource-container-id="366411874" data-linked-resource-container-version="2" data-media-id="62d16282-66fc-4add-ae1c-10b294ccdbf8" data-media-type="file" alt="git pull" /></span>

Okay, we've seen how to pull/download changes from the remote. On the other hand, how about sending local changes to remote?

### 🔵 Upload to remote

Git provides a **porcelain** command called `git push`:

👉 **git push**

Performing `git push origin main` will first upload the changes to remote:

### 🔵 Giving immutable names to commits

Until now, we learned that **branches are simply mutable references to commits**, that's why we can move a branch pointer at any time.

However, Git also provides a way to *give immutable references*, which cannot have their pointers changed (unless you delete them and create them again).

Immutable references are helpful when we want to *label/mark commits* that are ready for some production release, for example.

Yes, we are talking about **tags**.

👉 **git tag**

Using the porcelain `git tag` command, we can give names to commits but we cannot perform reset or any other command which would change the pointer.

<span class="confluence-embedded-file-wrapper image-center-wrapper"><img src="./media/attachments/366411874/383779071.png" class="confluence-embedded-image image-center" loading="lazy" data-image-src="attachments/366411874/383779071.png" data-height="415" data-width="880" data-unresolved-comment-count="0" data-linked-resource-id="383779071" data-linked-resource-version="1" data-linked-resource-type="attachment" data-linked-resource-default-alias="pm62925zwulfu2vijjie.png" data-base-url="https://never4got10.atlassian.net/wiki" data-linked-resource-content-type="image/png" data-linked-resource-container-id="366411874" data-linked-resource-container-version="2" data-media-id="2fe14c20-2362-4f7e-96ac-f6203f56f9c5" data-media-type="file" alt="git tag" /></span>

It's quite useful for release versioning. Tags are located in the `.git/refs/tags` folder:

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ find .git/refs

...
.git/refs/tags
.git/refs/tags/v1.0
```

</div>

</div>

*If we want to change the tag pointer, we must delete it and create another one with the same name.*

## 💡 Git reflog

Last but not least, there's a command called `git reflog` which keeps all the changes we've made in our local repository.

<div class="code panel pdl" style="border-width: 1px;">

<div class="codeContent panelContent pdl">

``` syntaxhighlighter-pre
$ git reflog

369cd96 (HEAD -> fix, test) HEAD@{0}: reset: moving to main
b77b454 (main) HEAD@{1}: reset: moving to b77b
369cd96 (HEAD -> fix, test) HEAD@{2}: checkout: moving from main to fix
369cd96 (HEAD -> fix, test) HEAD@{3}: checkout: moving from fix to main
369cd96 (HEAD -> fix, test) HEAD@{4}: checkout: moving from main to fix
369cd96 (HEAD -> fix, test) HEAD@{5}: checkout: moving from fix to main
369cd96 (HEAD -> fix, test) HEAD@{6}: checkout: moving from main to fix
369cd96 (HEAD -> fix, test) HEAD@{7}: checkout: moving from test to main
369cd96 (HEAD -> fix, test) HEAD@{8}: checkout: moving from main to test
369cd96 (HEAD -> fix, test) HEAD@{9}: checkout: moving from test to main
369cd96 (HEAD -> fix, test) HEAD@{10}: commit: add files
b77b454 (main) HEAD@{11}: commit: another commit
5ea578a HEAD@{12}:
```

**It's quite useful** if we want to **go back and forth** on the Git timeline. Along with *reset, cherry-pick and similar*, it's a powerful tool if we want to master Git.
