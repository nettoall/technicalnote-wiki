# 쿠버네티스 기본 사용법

## POD 생성

- kubectl 명령어로 pod 생성 가능

- kubectl run \[POD 이름\] --image=\[생성할이미지이름\]

  - ex) kubectl run nginx-pod --image=nginx

- pod 생성 확인

  - kubectl get pod

- kubectl create와의 차이

  - 실행 명령 : kubectl create deployment dpy-nginx --image=nginx

  - deployment는 pod 관리 그룹

  - create로 생성하면 deployment라는 관리 그룹 내에 pod를 생성

  - run는 pod 만 생성

## 오브젝트

- pod, deployment 는 spec과 status 를 가짐

- 쿠버네티스에서는 spec과 status를 가진 pod, deployment를 object 라 부름

- 쿠버네티스에는 여러 유형의 object가 존재

  - Pod : 쿠버네티스에서 실행되는 최소 단위

    - 하나의 Pod는 1개 이상의 컨테이너를 가짐

  - Namespaces : 쿠버네티스 클러스에서 사용되는 리소스들을 구분해 관리하는 그룹

    - 기본 할당되는 defulat

    - 쿠버네티스 시스템에서 사용되는 kube-system

    - 온프레미스에서 쿠버네티스 사용할 경우, 외부에서 쿠버네티스 클러스터 내부로 접속하게 도와주는 컨테이너들이 속해 있는 metalib-systme

  - Volume : Pod가 생성될 때 Pod에서 사용할 수 있는 디렉토리 제공

    - Pod가 영속적인 개념이 아니 듯 제공되는 디렉토리도 임시로 사용

    - 영구 저장이 가능한 디렉토리를 volume 오브젝트를 통해 생성 가능

  - Service : Pod는 클러스터 내에서 유동적이므로 접속 정보가 고정일 수 없음

    - Pod 접속을 안정적으로 유지하도록 내/외부 연결

## Deployment

- 기본 Object 만으로는 한계가 있어 이를 좀 더 효율적으로 작동하도록 기능 조합, 추가하고 구현 한 것 : deployment

- deployment와 같은 목적으로 DaemonSet, CinfigMap, ReplicaSet, PersistentVolume, PersistentVolumeClain, StatefulSet 등의 추가 Object가 있음

- Deployment : Pod 에 기반을 둔, 레플리카셋 오브젝트를 합쳐 놓은 형태 (레플리카셋은 레플리케이션컨트롤러가 발전한 형태)

- 용어나 구조에 익숙해져야 함

- replicaset 동일한 pod 개수를 지정해서 생성할 수 있음 → 다 수의 pod를 만드는 replicaset object

- replicaset은 pod 수를 보장하는 기능만 제공하며 롤링 업데이트 기능 등이 추가된 deployment를 사용해 pod 수를 관리하기를 권장 → deployment는 replicaset를 관리하는 기능이 있다는 말?

### yaml 파일로 Deployment 로 pod 여러 개 생성하기

- kubectl create deployment 수행 시 replicas을 지정할 수 있는 방법은?

  - 초기에 replicas를 지정할 수 없나? → create 명령은 replicas 옵션 사용 못함

  - 현재로서는 pod를 생성하고 scale 명령을 사용하여 다수의 pod를 생성해야 함.

  - 이런 설정을 적용하려면 필요한 내용을 파일로 작성해야 함 → 오브젝트 스펙

  - 오브젝트 스펙은 yaml로 작성

  - 3개의 nginx pod를 deployment 오브젝트로 생성하는 yaml 파일 정의(ch3/3.2.4)\

### POD 자동 장애 처리 실습

1.  실행중인 POD의 IP 확인

    - kubectl get pods -o wide

2.  nginx 파드 컨테이너의 쉘에 접속

    - kubectl exec -it nginx-pod – /bin/bash

    - i(stdin : standard input 표준입력), t (tty) 표준 입력을 명령줄 인터페이스로 작성

    - \- - : exec에 대한 인자 값을 나누고 싶을 때 사용 → exec의 옵션으로 인식 할 수 있으므로 – 옵션을 주어 exec의 옵션이 아님을 인지시킴

3.  nginx의 pid 확인

    - cat /run/nginx.pid

4.  프로세스 생성된 시간 확인

    - ls -l /run/nginx.pid

5.  master node 에서 nginx-pod의 ip 에서 돌아가는 웹페이지를 1초마다 한번씩 요청하는 스크립트 실행

    - i=1; while true; do sleep 1; echo \$((i++)) `` `curl --silent [pod ip] | grep title ``\`; done

6.  nginx pod 컨테이너에서 nginx 프로세서 pid 를 kill 명령으로 종료

7.  master node 에서 1초마다 요청하는 상태를 조회하여 kill 이후 자동으로 pod가 살아나는지 확인

### POD 업데이트 실패시 복구

- 업데이트 실패시 ( 예 : 잘못된 버전 입력 등) 복구

1.  set image 명령으로 nginx 의 버전을 잘못 입력(1.17.23 으로)

    - kubectl set image deployment rollout-nginx nginx=nginx:1.17.23 --record

2.  pod 상태 확인하기

    - kubectl get pods -o=custom-columns=NAME:.metadata.name,IP:.status.podIP,STATUS:.status.phase,NODE:.spec.nodeName

    - STATUS가 Pending 상태 지속됨

3.  rollout status 명령으로 상태 확인

    - kubectl rollout status deployment rollout-nginx

    - =\> Waiting for deployment “rollout-nginx” rollout to finish: 1 out of 3 new replicas have been updated… 출력 후 진행 안 됨 (Ctrl + c 로 빠져나옴)

4.  describe 명령으로 상태를 상세히 확인

    - kubectl describe deployment rollout-nginx

    - replicas가 생성되는 과정에서 멈춤

5.  업데이트할때 명령을 rollout history로 확인

    - kubectl rollout history deployment rollout-nginx

    - 마지막 단계(revision 버전 최종 단계)가 nginx 1.17.23 업데이트 명령

6.  rollout undo 명령으로 실행을 취소

    - kubectl rollout undo deployment rollout-nginx

7.  pod 상태 확인

    - kubectl get pods -o=custom-columns=NAME:.metadata.name,IP:.status.podIP,STATUS:.status.phase,NODE:.spec.nodeName

8.  rollout history 로 실행된 명령들 확인

    - kubectl rollout history deployment rollout-nginx

    - 5번에서 보았던 마지막 단계가 아닌 바로 이전 단계가 최종 단계(revision) 으로 변경됨

9.  curl 명령으로 nginx 서버 버전 확인

    - curl -I --silent 192.16.221.146 \| grep Server

10. rollout status 명령으로 변경이 정상 적용 확인

    - kubectl rollout status deployment rollout-nginx

11. describe 명령으로 상태 확인

    - kubectl describe deployment rollout-nginx

### 특정 시점으로 POD 복구하기

- Rollout undo 명령과 --to-revision 옵션을 주어 특정 시점의 revision 으로 복구

1.  처음 상태인 revision 1로 돌아감

    - kubectl rollout undo deployment rollout-nginx --to-revision=1

    - 돌아갈 revision을 사전에 알고 있어야 한다.

    - kubectl rollout history deployment rollout-nginx 로 명령어를 조회 가능해야 함

    - 새로운 pod를 생성함
